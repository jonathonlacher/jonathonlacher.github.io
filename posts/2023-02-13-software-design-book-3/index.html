Book notes: A Philosophy of Software Design, Chapters 8-11

Jan 19, 2023 DRAFT

Chapter 8: Pull Complexity Downwards

  1. As a module developer, you should strive to make life easy as possible for the users of your module, even if it means extra work for you. / It is more important for a module to have a simple interface than a simple implementation. (55)
  2. Example: Config parameters
    1. Config parameters provide an easy excuse to avoid dealing with important issues and pass them off to someone else. (57)
    2. Avoid config params as much as possible. Ask yourself, will users or higher-level modules be able to determine a better value? (57)
    3. See if you can compute reasonable defaults automatically. (57)
  3. Pulling complexity down makes the most sense if:
    1. The complexity is closely related to the class’s existing functionality. (57)
    2. Results in many simplifications elsewhere in the application. (57)
    3. Simplifies the interface of the class. (57)

Chapter 9: Better Together or Better Apart?

  1. When deciding to combine or separate things, the goal is to reduce compexity of the system as a whole and improve its modularity. It may appear the best way to do this is to divide (59)
  2. However, the act of subdividing creates additional complexity that was not present before the subdivision. (59)
  3. Keep them together if:
    1. Information is shared between them. (61)
    2. It simplifies the interface. (61)
    3. It eliminates duplication. This aproach is most effecive if repeated code is long and the replacement has a simple signature. (61)
  4. Separate them if:
    1. If it’s special-purpose code associated with a general purpose mechanism. If code is repeated over and over, this is a red flag that you haven’t found the right abstractions. (62)
